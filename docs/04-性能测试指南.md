## 二、测试框架设计哲学

### 2.1 为什么不用现成工具（wrk / ab / k6）？

| 考虑因素 | 通用工具的问题 | 自研工具的优势 |
|---------|--------------|--------------|
| **协议支持** | wrk 只支持 HTTP；WebSocket 需要 k6 等，学习成本高 | 原生支持项目的 HTTP API + WebSocket + 业务流程 |
| **业务语义** | 通用工具只能发原始请求，不理解"登录→建连→发消息"的业务流 | 内置注册/登录/握手/发消息的完整链路 |
| **协议细节** | 无法精确控制 WebSocket 帧的 mask、分帧、payload 格式 | 完全掌控帧构造，可以测试各种边界情况 |
| **瓶颈定位** | 看不到 EAGAIN 次数、连接建立时间等内部细节 | 暴露 EAGAIN、bytes_sent/recv、连接成功率等细粒度指标 |
| **面试加分** | "我用 wrk 测的" | "我自己写的 epoll 异步压测工具，支持 HTTP 流水线" |

### 2.2 两层架构思路

```
                    ┌─────────────────────────────────┐
                    │        功能验证层                │
                    │   ws_client_cli / load_tester    │
                    │   (阻塞 IO, 简单直观)            │
                    └───────────────┬─────────────────┘
                                    │ 确认功能正确后
                                    ▼
                    ┌─────────────────────────────────┐
                    │        极限压测层                │
                    │  async_ws_bench / http_bench     │
                    │  (epoll 事件驱动, 榨干性能)      │
                    └─────────────────────────────────┘
```

**先验证正确性，再追求极限性能** — 这是工程实践的基本准则。如果功能都不对，压出来的 QPS 毫无意义。

### 2.3 公共库复用设计

`ws_client_lib` 抽取了所有工具共用的基础能力：

```cpp
// ws_client_lib.h 提供的公共接口
struct Config { ... };                           // 统一配置结构
bool ParseArgs(int argc, char* argv[], Config*); // 命令行解析
bool HttpPostLogin(const Config&, ...);          // HTTP 登录
int ConnectTcp(const std::string& host, int port);  // TCP 连接
std::string BuildClientWebSocketTextFrame(...);  // WebSocket 帧构造
bool WebSocketHandshake(int fd, ...);            // WebSocket 握手
void ReceiveLoop(int fd);                        // 接收循环
```

这样做的好处：
1. **DRY 原则** — 登录、握手等逻辑只写一次
2. **一致性** — 所有工具使用同一套协议实现，不会因为实现差异导致测试结果不可比
3. **可维护性** — 协议变更时只改一处

## 三、工具详解：http_bench（HTTP 推送压测）

### 3.1 核心架构

```
main()
  │
  ├── 创建 N 个工作线程
  │     每个线程：
  │     ├── epoll_create1() 创建 epoll 实例
  │     ├── 创建 M 个 TCP 连接 (非阻塞)
  │     ├── epoll 事件循环：
  │     │     ├── EPOLLOUT → 填充流水线 + 发送请求
  │     │     ├── EPOLLIN  → 接收响应 + 解析 + 续填流水线
  │     │     └── 所有消息发完且响应收完 → 关闭连接
  │     └── close(epfd)
  │
  └── 主线程：每 500ms 打印进度 (QPS, 错误数, 活跃连接)
```

### 3.2 关键技术：HTTP 流水线（Pipelining）

```
传统模式（串行）：           流水线模式（pipeline_depth=3）：
 
Client   Server              Client   Server
  │─ Req1 ─→│                  │─ Req1 ─→│
  │←─ Resp1─│                  │─ Req2 ─→│
  │─ Req2 ─→│                  │─ Req3 ─→│  ← 连续发 3 个不等回复
  │←─ Resp2─│                  │←─ Resp1─│
  │─ Req3 ─→│                  │←─ Resp2─│
  │←─ Resp3─│                  │←─ Resp3─│
                               │─ Req4 ─→│  ← 收到 Resp1 后补发
  总时间 = 3 × RTT             总时间 ≈ 1 × RTT + 传输时间
```

**为什么用流水线？**

HTTP/1.1 Keep-Alive 虽然复用了 TCP 连接，但默认是"请求-响应-请求-响应"串行模式。对本地测试来说 RTT 很低，但服务端处理有延迟，串行模式下客户端大部分时间在**等**。流水线让客户端一次性发出多个请求，服务端按顺序返回响应，极大减少了客户端等待时间。

代码中的实现：

```cpp
// 流水线的核心逻辑
void FillSendBuffer(Connection* conn, const Config& cfg) {
    int can_send = conn->pipeline_depth - conn->pending_responses;  // 还能发几个
    int remaining = conn->messages_to_send - conn->messages_sent;   // 还剩几个
    int to_send = std::min(can_send, remaining);                    // 取较小值
    
    for (int i = 0; i < to_send; ++i) {
        conn->send_buffer += BuildHttpRequest(...);  // 拼接到发送缓冲区
    }
    conn->messages_sent += to_send;
    conn->pending_responses += to_send;
}
```

`pipeline_depth` 的含义是**最多同时在途的请求数**。设为 10 意味着最多有 10 个请求已发出但尚未收到响应。这个值需要权衡：
- **太小**（如 1）→ 退化成串行模式
- **太大**（如 1000）→ 服务端积压过多请求，可能 OOM 或超时
- **推荐值**：10~50，根据服务端处理能力调整

### 3.3 关键技术：epoll 事件驱动

为什么用 epoll 而不是每个连接一个线程？

| 模式 | 100 连接 | 10,000 连接 |
|------|---------|------------|
| 线程模型 | 100 线程，勉强可用 | 10,000 线程，频繁上下文切换，OS 直接崩 |
| epoll | 1 线程管理 100 fd | 1 线程管理 10,000 fd，O(1) 事件通知 |

代码中的关键设置：

```cpp
// 非阻塞 socket
SetNonBlocking(fd);

// TCP_NODELAY 禁用 Nagle 算法 — 小包立即发送，不凑包
setsockopt(fd, IPPROTO_TCP, TCP_NODELAY, &flag, sizeof(flag));

// 增大发送缓冲区 — 减少内核态缓冲区满导致的 EAGAIN
int sndbuf = 256 * 1024;
setsockopt(fd, SOL_SOCKET, SO_SNDBUF, &sndbuf, sizeof(sndbuf));
```

### 3.4 响应解析

```cpp
int ParseResponses(Connection* conn) {
    while (true) {
        // 1. 找到 \r\n\r\n（HTTP 头结束标志）
        size_t header_end = conn->recv_buffer.find("\r\n\r\n");
        if (header_end == std::string::npos) break;
        
        // 2. 解析 Content-Length
        // ...
        
        // 3. 检查 body 是否收全
        size_t total_len = header_end + 4 + content_length;
        if (conn->recv_buffer.size() < total_len) break;
        
        // 4. 移除已处理的响应
        conn->recv_buffer.erase(0, total_len);
        count++;
    }
    return count;
}
```

这里的难点是**TCP 粘包**：多个 HTTP 响应可能在一次 `recv()` 中全部收到，也可能一个响应分多次收到。必须通过 `Content-Length` 精确切割。

## 四、工具详解：async_ws_bench（WebSocket 异步压测）

### 4.1 核心架构

```
main()
  │
  ├── [阶段 1] 串行注册 + 登录，获取 tokens[]
  │
  ├── [阶段 2] 启动 W 个 Worker 线程
  │     每个 Worker：
  │     ├── epoll_create1()
  │     ├── 创建 N/W 个 AsyncClient
  │     │     每个 Client：
  │     │     ├── 非阻塞 connect()
  │     │     ├── WebSocket 握手
  │     │     ├── 预生成所有消息帧
  │     │     └── 批量填充 + 发送
  │     └── epoll 事件循环 (EPOLLET 边缘触发)
  │
  └── 主线程：实时统计 (发送/接收 QPS, EAGAIN 次数)
```

### 4.2 与 http_bench 的核心区别

| 特性 | http_bench | async_ws_bench |
|------|-----------|----------------|
| 协议 | HTTP/1.1 | WebSocket |
| 测试链路 | 客户端 → Logic HTTP → Kafka → ... | 客户端 → Comet WS → Logic gRPC → Kafka → ... |
| 连接建立 | 直接 TCP | TCP + WebSocket 握手 |
| 认证方式 | HTTP Header Token | WebSocket URL Query Token |
| 帧格式 | HTTP 请求/响应 | WebSocket 帧（mask + opcode） |
| 用户系统 | 手动指定 from/to | 自动注册 + 登录 |

### 4.3 关键技术：帧预生成

```cpp
void PrepareMessages(int count, int to_user_id, int room_id) {
    msg_frames_.reserve(count);
    
    for (int i = 0; i < count; ++i) {
        nlohmann::json j;
        j["type"] = "single_chat";
        j["to_user_id"] = to_user_id;
        j["content"] = {{"text", "b"}};  // 最小 payload
        
        // 预先完成 JSON 序列化 + WebSocket 帧构造 + mask
        msg_frames_.push_back(BuildWebSocketFrame(j.dump()));
    }
}
```

**为什么要预生成？**

在热循环中做 JSON 序列化 + WebSocket 帧构造会引入大量 CPU 开销：
- `nlohmann::json` 的 `dump()` 涉及内存分配和字符串拼接
- `BuildWebSocketFrame()` 需要计算长度、生成 mask、逐字节异或

如果每发一条消息才临时构造，测出来的 QPS 反映的是 **"客户端 JSON 构造速度"** 而不是 **"服务端处理能力"**。预生成把这些开销挪到测试开始前，热路径只有纯粹的 `send()` 系统调用。

### 4.4 关键技术：批量发送

```cpp
bool OnWritable() {
    // ... 先发完缓冲区中的数据 ...
    
    // 批量填充发送缓冲区（最多 64KB）
    const size_t kMaxBatch = 64 * 1024;
    while (msg_sent_ < msg_to_send_ && send_buf_.size() < kMaxBatch) {
        send_buf_ += msg_frames_[msg_sent_];
        msg_sent_++;
    }
    
    // 立即尝试发送
    return OnWritable();  // 递归，直到 EAGAIN 或全部发完
}
```

**为什么限制 64KB？**

- 太小 → 频繁触发 `send()` 系统调用，syscall 开销大
- 太大 → 一次性占用大量用户态内存，可能导致 `send()` 长时间阻塞在内核拷贝
- 64KB 是经验值，与典型的 TCP socket 发送缓冲区大小相当

### 4.5 关键技术：EPOLLET 边缘触发

```cpp
ev.events = EPOLLIN | EPOLLOUT | EPOLLET;  // 注意 EPOLLET
```

**边缘触发 vs 水平触发：**

| 触发模式 | 行为 | 优点 | 缺点 |
|---------|------|------|------|
| LT（水平触发） | 只要 fd 就绪就一直通知 | 简单安全 | 可能频繁唤醒 epoll_wait |
| ET（边缘触发） | 只在状态**变化**时通知一次 | 减少 epoll_wait 调用次数 | 必须一次读/写到 EAGAIN |

async_ws_bench 使用 ET 是因为：
1. 每个 Client 在可写时会循环发送直到 EAGAIN，符合 ET 的使用要求
2. 高连接数时减少了 epoll_wait 的惊群效应
3. 代码中额外有**主动触发写**的逻辑来弥补 ET 可能遗漏的事件：

```cpp
// 主动触发写（处理 EAGAIN 后的重试）
for (auto& kv : clients) {
    AsyncClient* c = kv.second.get();
    if (c->HasPendingWrite() && c->state() >= AsyncClient::kConnected) {
        c->OnWritable();
    }
}
```

### 4.6 WebSocket 帧构造细节

```cpp
std::string BuildWebSocketFrame(const std::string& payload) {
    std::string frame;
    frame.push_back(0x81);  // FIN=1 (完整帧), opcode=0x1 (文本)
    
    // 长度编码：7 位 / 7+16 位 / 7+64 位
    if (len < 126) {
        frame.push_back(0x80 | len);     // MASK=1
    } else if (len < 65536) {
        frame.push_back(0xFE);           // 0x80 | 126
        frame.push_back(len >> 8);
        frame.push_back(len & 0xFF);
    }
    
    // 客户端到服务端必须 mask（RFC 6455 规定）
    uint32_t mask = RandomMask();
    frame.append(mask_bytes, 4);
    
    // payload 逐字节异或 mask
    for (size_t i = 0; i < len; ++i)
        frame.push_back(payload[i] ^ mask_bytes[i % 4]);
}
```

**为什么客户端必须 mask？**

RFC 6455 §5.3 规定客户端发往服务端的帧**必须**使用 masking，防止中间缓存代理将 WebSocket 流量误识别为 HTTP 缓存内容（一种安全攻击向量）。服务端发往客户端的帧则**不需要** mask。

## 五、工具详解：load_tester（多模式压测）

### 5.1 三种模式对比

```
Mode::kWsSend        → 压测 WebSocket 消息发送全链路
Mode::kHttpHistory   → 压测 HTTP 历史消息查询
Mode::kBizFlow       → 压测业务流程（注册 → 登录 → 建房 → 加入）
```

| 模式 | 测试目标 | 覆盖组件 | 适合观察 |
|------|---------|---------|---------|
| ws_send | 消息推送主链路 | Comet + Logic + Kafka + Job | 端到端延迟、消息吞吐 |
| http_history | 历史消息读取 | Logic + MySQL + Redis | 查询性能、缓存命中率 |
| biz_flow | 业务流程 | Logic + MySQL + Redis | 数据库写入能力、事务性能 |

### 5.2 ws_send 模式的实现

```cpp
// 每个线程的工作函数
auto worker_fn = [&](int idx_begin, int idx_end) {
    for (int i = idx_begin; i < idx_end; ++i) {
        // 1. 登录获取 token
        HttpPostLogin(base_cfg, &user_id, &token);
        
        // 2. TCP 连接 + WebSocket 握手
        int fd = ConnectTcp(comet_host, comet_port);
        WebSocketHandshake(fd, host, port, token);
        
        // 3. 循环发送消息
        for (int m = 0; m < messages_per_conn; ++m) {
            json j = { {"type", "single_chat"}, ... };
            string frame = BuildClientWebSocketTextFrame(j.dump());
            send(fd, frame.data(), frame.size(), 0);
            sleep_for(milliseconds(send_interval_ms));  // ← 注意：有间隔
        }
    }
};
```

**与 async_ws_bench 的本质区别**：load_tester 使用**阻塞 IO + sleep 间隔**，模拟的是真实用户行为（人不可能每秒发上万条消息），适合中低负载下验证功能正确性。async_ws_bench 则去掉了一切开销，追求极限吞吐。

### 5.3 http_history 模式

```cpp
// 多线程循环调用 /api/session/history
for (int i = 0; i < messages_per_conn; ++i) {
    json body = {
        {"session_id", "single:1:2"},
        {"anchor_seq", 0},
        {"limit", 20}
    };
    HttpPostJsonRaw(cfg, "/api/session/history", body, &resp);
}
```

这个模式测的是**读性能**，重点观察：
- MySQL 慢查询日志中是否出现历史查询
- Redis 缓存命中率
- Logic 的 HTTP 线程池是否成为瓶颈

### 5.4 biz_flow 模式

```
每个虚拟用户的一轮操作：
  ① POST /api/register      → 注册（409 = 已存在，视为成功）
  ② POST /api/login          → 登录，获取 token
  ③ POST /api/admin/chatroom/create → 创建聊天室
  ④ POST /api/chatroom/join  → 加入聊天室
```

测的是**写密集型业务链路**的性能，涉及多次 MySQL INSERT/UPDATE 和 Redis SET/HSET。

## 六、工具详解：ws_client_cli（功能验证客户端）

### 6.1 使用场景

```bash
# 单聊模式：以 user1 身份向 user_id=2 发消息
./ws_client_cli --account user1 --password 123456 --to-user 2

# 聊天室模式：以 user1 身份在 room 1001 中发消息
./ws_client_cli --account user1 --password 123456 --room-id 1001
```

启动后进入交互模式，每输入一行文本就发送一条消息，同时打印收到的下行消息。

### 6.2 为什么需要这个工具

在做任何性能测试之前，**必须先用交互式客户端确认功能正确**。否则可能出现：
- 消息发出去了但服务端直接丢弃（路由错误）→ QPS 虚高
- WebSocket 握手失败 → 压测工具报告 0 连接成功但你以为是性能问题
- 消息格式不对 → 服务端 400 但你看到的是"响应收到了"以为成功

## 七、为什么这么写：关键设计决策

### 7.1 为什么用原始 socket 而不用 libcurl / Boost.Beast？

```cpp
// 手动构造 HTTP 请求
std::ostringstream req;
req << "POST " << endpoint << " HTTP/1.1\r\n"
    << "Host: " << host << ":" << port << "\r\n"
    << "Content-Type: application/json\r\n"
    << "Content-Length: " << body_str.size() << "\r\n"
    << "Connection: keep-alive\r\n"
    << "\r\n"
    << body_str;
```

**原因：**
1. **性能开销最小化** — libcurl 内部有很多通用逻辑（重定向处理、cookie 管理、SSL 等），压测工具不需要这些
2. **精确控制** — 需要实现 HTTP 流水线，libcurl 默认不支持（HTTP/1.1 pipelining 在 libcurl 中被标记为实验性功能）
3. **零依赖** — 压测工具只依赖 POSIX socket API 和 nlohmann/json，编译简单
4. **面试考点** — 体现对 HTTP 协议的深入理解

### 7.2 为什么每个连接用不同的 from_user_id？

```cpp
conns[i].from_user_id = cfg.from_user_id + conns_start + i;
```

如果所有连接都用同一个 `from_user_id`，服务端可能：
- 触发限流（同一用户发消息频率限制）
- 共享同一个 session，无法测试并发 session 处理能力
- Redis INCR 的 key 都是同一个 session，变成串行瓶颈

用不同的 `from_user_id` 可以模拟真实的多用户场景，充分暴露服务端的并发处理能力。

### 7.3 为什么 async_ws_bench 要预注册用户？

```cpp
// 1. 串行注册 + 登录
for (int i = 0; i < connections; ++i) {
    HttpRegister(logic_host, logic_port, acc, password);
    HttpLogin(logic_host, logic_port, acc, password, &uid, &tok);
    tokens.push_back(tok);
}

// 2. 用已有的 tokens 做异步压测
for (auto& w : ws) w->Start();
```

**原因：**
- 注册/登录涉及 MySQL 写入，很慢（几 ms 到几十 ms）
- 如果把注册/登录混在压测热路径中，测出来的 QPS 反映的是 MySQL 写入速度，而不是消息推送速度
- **分离准备阶段和压测阶段**是性能测试的基本方法论

### 7.4 为什么使用 `std::atomic` 做统计而不是 mutex？

```cpp
std::atomic<int64_t> g_requests_sent{0};
std::atomic<int64_t> g_responses_received{0};
std::atomic<int64_t> g_errors{0};
```

**原因：**
- `atomic` 的 `fetch_add` 操作在 x86 上编译为 `lock xadd` 指令，大约 10~20ns
- `mutex` 的 `lock()/unlock()` 在无竞争时约 25ns，有竞争时可达数 μs
- 压测工具每发一条消息就要更新计数器，如果计数器本身成为瓶颈，就测不出服务端的真实性能

### 7.5 为什么 payload 用 `{"text": "b"}` 而不是真实内容？

```cpp
j["content"] = {{"text", "b"}};  // 最小 payload
```

性能测试的目标是测**消息处理链路的吞吐**，不是测**网络带宽**。最小 payload 可以：
1. 减少网络层的影响（带宽不成为瓶颈）
2. 减少 JSON 序列化/反序列化时间
3. 让瓶颈集中在 Logic/Redis/Kafka 等处理层

如果需要测带宽或大消息场景，可以改为更长的 payload。

### 7.6 为什么 http_bench 的响应解析这么"简陋"？

```cpp
// 只解析 Content-Length，不看 Transfer-Encoding、不看状态码
size_t cl_pos = header.find("Content-Length:");
```

这不是 Bug，而是**有意为之**。压测工具的响应解析越轻量，客户端的 CPU 开销越小，越能把压力完整传递给服务端。完整的 HTTP 解析（处理 chunked encoding、压缩、3xx 重定向等）会增加客户端 CPU 负担，影响 QPS 测量的准确性。

因为我们知道服务端只会返回固定格式的 JSON 响应（有 `Content-Length`，不用 chunked），所以可以安全地简化解析。

## 八、如何测出真正的性能

### 8.1 测试环境准备

#### 系统参数调优

```bash
# 1. 增大文件描述符限制（每个连接消耗一个 fd）
ulimit -n 100000

# 2. 增大本地端口范围（压测客户端需要大量源端口）
echo "1024 65535" | sudo tee /proc/sys/net/ipv4/ip_local_port_range

# 3. 开启 TIME_WAIT 快速回收（压测结束后大量 TIME_WAIT）
echo 1 | sudo tee /proc/sys/net/ipv4/tcp_tw_reuse

# 4. 增大 TCP 连接队列（防止 SYN 洪水导致连接失败）
echo 65535 | sudo tee /proc/sys/net/core/somaxconn

# 5. 关闭 SWAP（防止内存不足时 swapping 导致延迟抖动）
sudo swapoff -a
```

#### 服务端准备

```bash
# 确保所有服务正常运行
# Logic (HTTP 9101 + gRPC 9100)
# Comet (WebSocket 9200 + gRPC 9205)
# Job (Kafka 消费)
# Redis、MySQL、Kafka 均正常
```

### 8.2 科学的测试流程

```
步骤 1: 功能验证
  └── ws_client_cli 手动发几条消息，确认端到端正常

步骤 2: 预热
  └── 用小负载（10 连接 × 100 消息）跑一次，让 JIT/缓存/连接池预热

步骤 3: 基准测试
  └── 固定参数跑 3 次，取中位数
  └── 每次之间等待 30 秒让系统恢复

步骤 4: 逐步加压
  └── 连接数: 10 → 50 → 100 → 200 → 500 → 1000
  └── 观察 QPS 增长是否线性，找到拐点

步骤 5: 持续压测
  └── 在最优连接数下持续跑 5 分钟
  └── 观察 QPS 是否稳定、内存是否泄漏
```

### 8.3 消除客户端瓶颈

**核心原则：确保压力来自服务端而非客户端。**

| 检查点 | 诊断方法 | 解决方案 |
|--------|---------|---------|
| 客户端 CPU 打满 | `top` 看压测进程 CPU | 增加线程数 / 用多台机器压 |
| 客户端网络带宽满 | `iftop` 看带宽 | 缩小 payload / 用多网卡 |
| 文件描述符不够 | 错误日志有 EMFILE | `ulimit -n 100000` |
| 端口耗尽 | 连接失败，`errno=EADDRNOTAVAIL` | 增大 port_range / 多 IP |
| EAGAIN 过多 | 统计面板的 EAGAIN 数 | 增大 SO_SNDBUF / 减少连接数 |

### 8.4 典型的测试命令

#### 场景 1：HTTP API 极限吞吐

```bash
./http_bench \
  --host 127.0.0.1 --port 9101 \
  --connections 200 --threads 4 \
  --messages 500 --pipeline 20 \
  --from-user 1 --to-user 2
```

**参数解释：**
- 200 个 TCP 连接分到 4 个线程（每线程 50 个连接）
- 每个连接发 500 条消息，流水线深度 20
- 总请求数 = 200 × 500 = 100,000

#### 场景 2：WebSocket 消息推送极限

```bash
./async_ws_bench \
  --logic-host 127.0.0.1 --logic-port 9101 \
  --comet-host 127.0.0.1 --comet-port 9200 \
  --connections 200 --workers 4 \
  --messages 1000 \
  --account bench_ --password bench123
```

**参数解释：**
- 200 个用户，自动注册 bench_0 ~ bench_199
- 每用户通过 WebSocket 发 1000 条消息
- 总消息数 = 200 × 1000 = 200,000

#### 场景 3：聊天室广播

```bash
./async_ws_bench \
  --logic-host 127.0.0.1 --logic-port 9101 \
  --comet-host 127.0.0.1 --comet-port 9200 \
  --connections 100 --workers 4 \
  --messages 500 --room-id 1001
```

**参数解释：**
- 100 个用户全部加入 room 1001
- 每用户发 500 条消息 → 每条消息广播给 99 个其他用户
- 总发送消息数 = 100 × 500 = 50,000
- 总接收消息数 ≈ 50,000 × 99 = 4,950,000

#### 场景 4：业务流程压测

```bash

  --mode biz_flow \
  --logic-host 127.0.0.1 --logic-port 9101 \
  --account stress_ --password test123 \

```

#### 场景 5：历史消息查询压测

```bash
./load_tester \
  --mode http_history \
  --logic-host 127.0.0.1 --logic-port 9101 \
  --history-session-id single:1:2 \
  --history-limit 50 \
  --io-threads 8 \
  --messages-per-conn 1000 --interval-ms 0
```

## 十、常见误区与陷阱

### 误区 1：在同一台机器上压测

**问题**：客户端和服务端争抢 CPU/内存/网络，测出来的不是服务端能力，而是整台机器的极限。

**建议**：生产级压测应该用独立的压测机。但对于项目练手/面试准备，同机压测可以接受，只需意识到**数字会偏低**。

### 误区 2：只看最终 QPS 不看曲线

**问题**：如果前 5 秒 QPS 为 50,000 但最后 5 秒降到 5,000（因为 Kafka 积压、MySQL 锁等待等），平均 QPS 好看但实际体验极差。

**建议**：观察实时输出中的 QPS 变化趋势。稳定的 30,000 QPS 比先高后低的"平均 40,000"更有价值。

### 误区 3：连接数越多越好

```
连接数 10   → QPS 5,000    (CPU 30%)
连接数 100  → QPS 40,000   (CPU 80%)
连接数 200  → QPS 50,000   (CPU 95%)  ← 最优点
连接数 500  → QPS 45,000   (CPU 100%) ← 开始下降！
连接数 1000 → QPS 30,000   (CPU 100%, 上下文切换爆炸)
```

**原因**：连接数超过 CPU 核心数的最优并发度后，线程切换、锁竞争、内存开销等副作用开始主导。需要**逐步加压**找到最优值。

### 误区 4：不区分"发送 QPS"和"端到端 QPS"

- **发送 QPS**：客户端每秒发出的消息数（只看发送端）
- **端到端 QPS**：消息从发出到对端收到的吞吐（看发送 + 接收 + 路由 + 存储完整链路）

async_ws_bench 同时统计 `msg_sent` 和 `msg_recv`，两者差距大说明有消息积压或丢失。

### 误区 5：忽略 EAGAIN

```
EAGAIN 次数: 150,000
```

EAGAIN 意味着 TCP 发送缓冲区满了，内核来不及把数据发出去。大量 EAGAIN 说明：
1. 客户端发送速度超过了网络带宽
2. 服务端处理不过来导致 TCP 窗口缩小
3. 需要调大 `SO_SNDBUF` 或减少并发连接数

## 十一、性能结果解读

### 11.1 关键指标说明

| 指标 | 含义 | 健康范围 |
|------|------|---------|
| QPS (发送) | 客户端每秒发出的消息/请求数 | 取决于服务端能力 |
| QPS (接收) | 客户端每秒收到的响应/消息数 | 应接近发送 QPS |
| 错误数 | 连接失败 + 发送失败 + 响应异常 | 应为 0 |
| EAGAIN 数 | send() 返回 EAGAIN 的次数 | 占发送总数 <10% |
| 连接成功率 | conn_established / conn_attempted | 应为 100% |
| 发送字节/接收字节 | 总吞吐量 | 用于计算网络利用率 |

### 11.2 项目实际性能数据

| 场景 | 配置 | QPS | 瓶颈 |
|------|------|-----|------|
| 单聊 (优化前) | 100 连接 | 8,330 | MySQL 同步写入 |
| 单聊 (MySQL 异步化) | 100 连接 | ~10,000 | Redis 路由查询 |
| 单聊 (+本地缓存) | 100 连接 | 16,656 | gRPC Unary 开销 |
| 单聊 (+gRPC 双向流) | 100 连接 | 33,322 | 并发度不足 |
| 单聊 (最终) | 200 连接 | **50,000** | Redis INCR（~14 万/秒理论值） |
| 聊天室 | 100 用户同房间 | **20,030** | 广播 fan-out |

### 11.3 瓶颈定位方法

```
QPS 不达预期？按以下顺序排查：

1. 看客户端 CPU → 如果打满，说明客户端是瓶颈，增加线程/机器
2. 看 EAGAIN 数 → 如果很多，说明网络是瓶颈
3. 看连接成功率 → 如果 <100%，检查 ulimit / somaxconn
4. 看 Redis QPS → `redis-cli info` 的 instantaneous_ops_per_sec
5. 看 Kafka lag → 消费组的消息积压量
6. 看 Logic/Comet 日志 → 是否有大量错误
7. 看 MySQL 慢查询 → 是否有阻塞写入
```
