# 消息全链路追踪：一条单聊消息的完整旅程

> 追踪用户 A（uid=1）发一条消息给用户 B（uid=2）的完整代码路径。
> 每一步标注 **文件名:行号**，可以直接跳到代码对照阅读。

## 总览

```
浏览器A → [WebSocket] → Comet → [gRPC双向流] → Logic → [Kafka] → Job
                                                                    ├→ PushHandler → [gRPC] → Comet → [WebSocket] → 浏览器B
                                                                    └→ PersistHandler → [MySQL INSERT]
```

---

## 第 1 步：浏览器发送 WebSocket 消息

**文件**: `web_demo/static/index.html`

```javascript
ws.send(JSON.stringify({
    type: "single_chat",
    to_user_id: 2,
    content: "你好",
    client_msg_id: "uuid-xxx"
}));
```

JSON 通过 WebSocket 协议封装成 Text Frame（opcode=0x1），经 TCP 发到 Comet 的 9200 端口。

---

## 第 2 步：Comet 收到 TCP 数据

**文件**: `comet/comet_server.cpp` L826-837

Muduo 的 EventLoop 检测到 socket 可读，触发 `OnMessage` 回调。此时连接已完成握手（`state == kOpen`），进入 `HandleWebSocketFrame`。

---

## 第 3 步：解析 WebSocket 帧

**文件**: `comet/comet_server.cpp` L542-599

1. 读取 opcode（L548-549）：确认是 Text Frame (0x1)
2. 读取 payload 长度（L551-571）：可能 7bit/16bit/64bit 三种编码
3. 读取 4 字节 masking key（L575-576）
4. XOR 解密 payload（L580-583）：`payload[i] = payloadData[i] ^ mask[i % 4]`
5. 根据 opcode 分发（L595-597）：`opcode == 0x1` 进入 `OnTextMessage`

**知识点**：RFC 6455 规定客户端到服务端必须 mask（防代理缓存中毒攻击）。

---

## 第 4 步：处理文本消息

**文件**: `comet/comet_server.cpp` L674-824

1. 解析 JSON，提取 type/to_user_id 等字段（L678-679）
2. 根据 `type == "single_chat"` 确定 `scene = "single"`（L689-696）
3. 构造 gRPC 请求 `UpstreamMessageRequest`（L758-774），填入 from_user_id、scene、to_user_id、content_json、comet_id、conn_id
4. 如果启用双向流模式（L776），通过 `SendToStream` 发送到 Logic；否则提交到线程池做同步 gRPC 调用（L800）

---

## 第 5 步（双向流模式）：通过持久流发送到 Logic

**文件**: `comet/comet_server.cpp` L330-344

```
SendToStream:
  1. NextRequestId() — 原子递增生成 request_id（L332）
  2. req_id % stream_count_ — 轮询分配到不同流（L336）
  3. 放入 send_queue，notify_one 唤醒写线程（L340-343）
```

**写线程** StreamWriterLoop（L271-301）：从队列取消息，注册回调到 `pending_callbacks_`，调用 `stream->Write()`。

**读线程** StreamReaderLoop（L304-328）：持续读取响应，通过 `request_id` 在 `pending_callbacks_` 中匹配回调并执行。

**为什么快**：传统模式每条消息都要创建 ClientContext + HTTP/2 HEADERS 帧；双向流只在建立时发一次 HEADERS，之后只有 DATA 帧。实测：16,656 -> 33,322 QPS（翻倍）。

---

## 第 6 步：Logic 处理上行消息

**文件**: `logic/grpc_service.cpp` L137-284

核心函数 `HandleUpstreamMessage`，分 5 个子步骤：

### 6.1 限流检查（L148-154）

Redis Sorted Set 滑动窗口算法，每用户每秒最多 20 条消息。
- `logic/redis_store.cpp` L434-481 `CheckRateLimit`

### 6.2 构造 session_id（L167-169，本地计算，0ms）

```
session_id = "single:" + min(1,2) + ":" + max(1,2) = "single:1:2"
```

A 发给 B 和 B 发给 A 产生相同的 session_id，不需要查数据库。

### 6.3 查询路由（L171-176）

调用 `RedisStore::GetUserRoutes`（`logic/redis_store.cpp` L101-135）：
1. 读锁查本地缓存（L108-116），命中则直接返回
2. 缓存 miss 则 Redis `HVALS route:user:2`（L137-167），取出所有 comet_id 并去重
3. 写锁更新缓存（L126-131），TTL = 1 秒

返回 `["comet-1"]`，表示用户 B 在 comet-1 上。

### 6.4 分配消息序号（L233-236）

调用 `RedisStore::IncrSessionMsgSeq`（`logic/redis_store.cpp` L411-432）：
- `INCR session_seq:single:1:2`，原子递增，返回如 42
- Redis 单线程模型保证原子性，这是当前性能瓶颈（约 5-7 万 INCR/秒）

### 6.5 发送到 Kafka（L251-268）

对每个目标 Comet 构造 `PushToCometRequest`，Protobuf 序列化后调用 `producer_->Send(comet_id, payload)`。

- Topic: `push_to_comet`
- Key: `comet_id`（同一 Comet 的消息进同一分区，保证有序）
- 如果接收方不在线（L270-283），comet_id 设为 `"persist_only"`，确保仍可持久化

---

## 第 7 步：Job 消费 Kafka — 推送路径

**文件**: `job/push_handler.cpp` L254-279

消费组 `meteor_push_push_group` 消费 `push_to_comet` topic。

1. 反序列化 Protobuf（L258）
2. 跳过 `persist_only` 消息（L266-268）
3. 流模式：放入队列，由 `StreamWriterLoop` 写入 gRPC 客户端流（L126-181）
4. 传统模式：提交到线程池，调用 `ProcessPushRequest`（L281-315），带指数退避重试（最多 3 次，100ms -> 400ms -> 1600ms），重试用尽记录 `[DLQ]` 死信日志

---

## 第 8 步：Comet 收到下行推送

**文件**: `comet/comet_grpc_service.cpp` L10-41

`CometServiceImpl::ProcessPushRequest`：
- 有 targets（单聊）：调用 `server_->PushToUsers(msg, users)`
- 有 room_id（聊天室/弹幕）：调用 `server_->PushToRoom(msg, room_id)`
- session_id == "broadcast"：调用 `server_->PushToAll(msg)`

---

## 第 9 步：发送给用户 B 的 WebSocket 连接

**文件**: `comet/comet_server.cpp` L346-370

`PushToUsers`：
1. 加锁查找 `user_conns_[uid]` 的所有连接，拷贝到局部 vector（L354-363）
2. 释放锁后逐个调用 `conn->send(frame)`（L364-368）

先拷贝再发送的原因：避免在持锁状态下执行可能阻塞的 send()，缩短临界区。

---

## 第 10 步：浏览器 B 收到消息

浏览器的 `ws.onmessage` 回调触发，解析 JSON 显示在聊天界面。

---

## 并行路径：Job 持久化

**文件**: `job/persist_handler.cpp` L77-89

另一个消费组 `meteor_push_persist_group` 独立消费同一 topic：

1. 反序列化（L81），检查 `need_persist`（L86）
2. 提交到线程池（L79），调用 `PersistMessage`（L92-149）
3. `mysql_real_escape_string` 转义内容（L119-121）
4. `INSERT ... ON DUPLICATE KEY UPDATE`（L124-128），唯一键 `(session_id, msg_seq)` 保证幂等
5. 失败重试（最多 3 次，指数退避），用尽记录 `[DLQ]` 日志

---

## 关键数据结构流转

| 阶段 | 格式 | 关键字段 |
|------|------|---------|
| 浏览器 -> Comet | WebSocket Text (JSON) | type, to_user_id, content |
| Comet -> Logic | gRPC UpstreamMessageRequest | from_user_id, scene, to_user_id, content_json |
| Logic 内部 | ChatMessage (Protobuf) | msg_id, session_id, msg_seq, content_json |
| Logic -> Kafka | PushToCometRequest 序列化 | comet_id, message, targets, need_persist |
| Job -> Comet | gRPC PushToCometRequest | 同上 |
| Comet -> 浏览器 | WebSocket Text (JSON) | msg_id, from_user_id, content, create_time |
| Job -> MySQL | SQL INSERT | session_id, msg_seq, content_json |

## 线程模型总结

| 组件 | 线程 | 职责 |
|------|------|------|
| Comet EventLoop 主线程 | 1 | 接受新连接 |
| Comet IO 工作线程 | N（comet_io_threads） | 处理 WebSocket 读写 |
| Comet gRPC 线程池 | 4（comet_grpc_pool_size） | 异步执行 gRPC 调用 |
| Comet 流写/读线程 | stream_count x 2 | 双向流读写 |
| Logic gRPC 服务线程 | gRPC 内部管理 | 处理 Comet 的 RPC |
| Logic HTTP 线程 | http_threads（默认 4） | HTTP API 请求 |
| Job Push 消费线程 | 1 | Kafka 消费循环 |
| Job Push RPC 线程池 | 4-8 | 传统模式 gRPC 推送 |
| Job Push 流写线程 | 每个 Comet 1 个 | 流模式写入 |
| Job Persist 消费线程 | 1 | Kafka 消费循环 |
| Job Persist 线程池 | 4-8 | MySQL 异步写入 |
